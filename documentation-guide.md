# 문서화 가이드

앱 프로젝트의 사용자/기여자/스토어용 문서를 일관되게 작성하기 위한 지침입니다. 이 가이드는 다른 앱 프로젝트에도 그대로 복사해 적용할 수 있도록 설계했습니다.

## 1. 소개

이 가이드는 다음 6종 문서의 목적, 포함 내용, 분량, 포맷을 정의합니다.

1. `overview.md` — 프로젝트 소개 (개발자/기여자용)
2. `user-guide.md` — 사용자 매뉴얼
3. `faq.md` — 자주 묻는 질문
4. `store-description.md` — 스토어 등록용 마케팅 카피
5. `updates.md` — 사용자용 릴리스 노트 (공개용)
6. `changelog.md` — 기술적 변경 기록 (내부용)

모든 문서는 `docs/app/` 폴더에 평면 배치합니다. `docs/` 루트에는 설계 메모·외부 도구 등 다른 문서들이 공존할 수 있으므로, 앱 제품 문서 6종과 본 가이드는 `docs/app/` 하위로 분리합니다.

## 2. 공통 원칙

### 언어
- 기본 언어는 **한국어**.
- `store-description.md`는 **한국어 + 영어 2종** 권장 (Play Store 다국어 등록 대비). 프로젝트에서 지원하는 다른 언어가 있으면 확장.
- `changelog.md`는 한국어 또는 영어 중 개발자가 편한 쪽.

### 파일명
- 소문자 + 하이픈. 예: `user-guide.md`, `store-description.md`.
- 확장자는 `.md`.

### 마크다운 스타일
- GitHub Flavored Markdown.
- H1은 문서당 1개(제목). 본문 섹션은 H2부터 시작.
- 이미지는 `docs/app/images/`에 저장, 상대 경로로 참조.
- 코드/UI 요소는 백틱으로 감싸기 (예: `설정 > 알림`).

### 자기완결성
- 각 문서는 **독립적으로 완결**되어야 합니다. 다른 앱 저장소의 문서나 `docs/` 루트의 설계 메모를 링크로 참조하지 않습니다.
- 같은 `docs/app/` 내 6종 문서 사이의 상호 참조는 허용 (예: overview의 "문서 맵" 표에서 user-guide.md 링크).
- 저장소 외부 문서·설계 메모·개발 노트로의 링크는 문서 본문에 포함하지 않습니다. 필요한 핵심 내용은 요약해 본 문서 내부에 옮겨 기입합니다.

### 버전·시간 표현
- "최근", "최신 버전" 같은 상대 표현 금지. 항상 **절대 버전·날짜**로 명시.
  - ❌ "최근 업데이트에서 위젯이 추가되었습니다"
  - ✅ "v7.0.0 (2026-04-23)에서 위젯이 추가되었습니다"

### 톤
- 사용자 대상 문서(`user-guide`, `faq`, `store-description`, `updates`): 경어체, 2인칭 ("~하세요").
- 개발자 대상 문서(`overview`, `changelog`): 평서체 허용.

### 업데이트 주기
| 문서 | 주기 |
|---|---|
| `changelog.md` | 릴리스마다 (개발 중에도 누적 가능) |
| `updates.md` | 릴리스 확정 시 |
| `store-description.md` | 기능 추가/제거 시 |
| `user-guide.md`, `faq.md`, `overview.md` | 분기 1회 + 주요 기능 변경 시 |

## 3. 문서 타입별 가이드

### 3.1 `overview.md`

- **목적:** 앱이 무엇이고, 왜 만들었으며, 누구를 위한 것인지 한눈에 전달.
- **독자:** 저장소를 처음 여는 개발자/기여자, 앱 맥락을 알고 싶은 사람.
- **포함할 내용:**
  - 앱 소개 1문단
  - 주요 기능 bullet
  - 기술 스택 요약
- **제외할 내용:** 상세 사용법, 스크린샷 시리즈, 변경 이력, 설계 메모·외부 문서 링크, 지원 환경·문서 맵 섹션.
  - 지원 플랫폼·최소 SDK·지원 언어 등은 `changelog.md`(내부용)나 `store-description.md`(사용자용)에서 다룹니다.
  - 문서 맵은 만들지 않습니다. `docs/app/` 폴더를 열면 파일 목록이 그대로 목차 역할을 합니다.
- **분량:** 300~500단어 (1~2 스크롤).
- **권장 섹션 순서:** 소개 → 주요 기능 → 기술 스택.

### 3.2 `user-guide.md`

- **목적:** 앱 설치 후 모든 기능을 사용할 수 있도록 안내.
- **독자:** 최종 사용자 (비개발자 포함).
- **포함할 내용:**
  - 설치 및 초기 설정
  - 기능별 사용법 (스크린샷 포함)
  - 권한 안내 (왜 필요한지, 어떻게 허용하는지)
  - 활용 팁
  - 간단한 트러블슈팅
- **제외할 내용:** 내부 아키텍처, 개발자용 설정, 미출시 기능.
- **분량:** 1500~3000단어 (기능 수에 비례).
- **권장 섹션 순서:** 시작하기 → 주요 기능별 사용법 → 설정 → 권한 → 자주 겪는 문제.

### 3.3 `faq.md`

- **목적:** 사용자 문의·리뷰에서 반복되는 질문에 빠르게 답변.
- **독자:** 문제를 겪고 있거나 기능 유무를 확인하려는 사용자.
- **포함할 내용:**
  - 질문-답변 쌍
  - 범주별 그룹핑 (일반 / 알림 / 워치 연동 / 데이터 등)
  - 답변당 3~5문장, 50~150단어
- **제외할 내용:** 튜토리얼성 장문 설명(→ `user-guide.md`), 버전별 변경(→ `updates.md`).
- **분량:** 전체 질문 10~25개.
- **권장 섹션 순서:** 범주별 H2 헤더 → 각 질문은 H3 또는 `**Q:** ... / **A:** ...` 패턴.

### 3.4 `store-description.md`

- **목적:** Play Store 등록 페이지에 바로 붙여넣을 마케팅 카피.
- **독자:** 앱 설치를 고민하는 잠재 사용자.
- **포함할 내용:**
  - 짧은 설명 (80자 이내)
  - 긴 설명 (4000자 이내)
  - 기능 하이라이트 bullet
  - 릴리스 노트 요약 (최신 버전)
  - 스크린샷 캡션
  - (선택) 검색 키워드
- **제외할 내용:** 기술 용어, 개발 이력, 내부 링크.
- **분량:** Play Store 한도 내. 한국어/영어 각 1세트.
- **권장 섹션 순서:** 짧은 설명 → 긴 설명 → 기능 하이라이트 → 릴리스 노트 → 스크린샷 캡션 → 키워드.

### 3.5 `updates.md`

- **목적:** 사용자 관점의 버전별 변경 사항 전달 (공개용).
- **독자:** 기존 사용자, 스토어 릴리스 노트 작성자.
- **포함할 내용:**
  - **모든 버전**의 섹션 — 버전 번호를 제목에 반드시 포함
  - 가능한 경우 릴리스 날짜 (git tag 날짜 기준). 없으면 빈칸 또는 생략
  - 사용자가 체감하는 변화만 선별
- **제외할 내용:** 커밋 해시, 내부 리팩터링, 개발자용 기술 상세, 미공개 변경.
- **분량:** 버전당 3~6 bullet. 최신 버전이 상단.
- **작성 규칙:**
  - "최신"·"이전 업데이트" 같은 **상대 표현은 제목에 쓰지 말 것**. 반드시 `## v{X.Y.Z}` 형태의 절대 버전 표기 사용.
  - 미출시/개발 중 버전은 `## v7.0.0 (개발 중)` 또는 `## v7.0.0 (준비 중)`으로 명시.
  - 날짜를 모르면 `## v6.0.0 — 2026-03-07`처럼 아는 것만 기입하고, 모르는 버전은 `## v2.3.0`로 날짜 없이 표기.
  - 내부 용어 금지. 예: "ProGuard 규칙 수정" ❌ → "앱 크래시 수정" ✅.
  - `History.md` 같은 원본 변경 이력이 없어 세부 내용이 불명인 경우, 해당 버전에 `- 세부 변경 내용 정보 없음`을 bullet으로 명시.
- **포맷 예시:**
  ```markdown
  ## v7.0.0 (개발 중)
  - 홈 화면 위젯 추가 — 2x2 / 4x4 크기로 타이머 제어 가능

  ## v6.0.1 — 2026-03-19
  - 워치 앱 안정성 개선

  ## v6.0.0 — 2026-03-07
  - 모바일 타이머 추가

  ## v2.3.0
  - 세부 변경 내용 정보 없음
  ```

### 3.6 `changelog.md`

- **목적:** 모든 릴리스의 변경 사항을 기술적 사실 중심으로 기록 (내부용).
- **독자:** 개발자, 기여자, 미래의 본인.
- **포함할 내용:**
  - **모든 버전**의 섹션 — 버전 번호를 제목에 반드시 포함
  - 가능한 경우 릴리스 날짜 (git tag 날짜 기준)
  - 리팩터링, 내부 개선, 빌드 변경 포함
  - 필요 시 관련 파일 경로, 이슈 번호 참조
- **제외할 내용:** 마케팅적 수사, 외부 설계 메모 링크.
- **분량:** 버전당 제한 없음. 최신 버전이 상단.
- **작성 규칙:**
  - "최신"·"이전 릴리스" 같은 **상대 표현 금지**. 반드시 `## v{X.Y.Z}` 절대 버전 표기.
  - 날짜를 모르면 버전만 기입.
  - `History.md` 등 원본 변경 이력이 없어 세부 내용이 불명인 버전은 `- 세부 변경 내용 정보 없음` 명시.
  - **변경 분류 헤더(`### 새 기능`, `### 개선`, `### 수정`, `### 내부`)는 사용하지 말 것.** 버전 섹션 아래에 변경 bullet만 평면 나열합니다.
- **포맷 예시:**
  ```markdown
  ## v7.0.0 (개발 중)
  - 모바일 위젯 (2x2, 4x4) — `app/src/main/java/.../widget/`
  - 디자인 리뉴얼 — 색상 체계 정리, 가독성 향상

  ## v6.0.1 — 2026-03-19
  - 워치 앱 안정성 개선

  ## v6.0.0 — 2026-03-07
  - 모바일 타이머
  - 워치페이스 컴플리케이션

  ## v2.3.0
  - 세부 변경 내용 정보 없음
  ```

## 4. 문서 간 관계 (중복 방지)

"같은 내용이면 한 곳에만" 원칙. 다른 문서에서는 링크 또는 요약으로만 참조합니다.

| 내용 유형 | 주 문서 | 링크/요약만 허용 |
|---|---|---|
| 앱 기능 사용법 | `user-guide.md` | `overview.md` (한 줄 요약), `faq.md` (문제 해결 중심) |
| 버전별 변경 (공개) | `updates.md` | `store-description.md` (최신만 요약) |
| 버전별 변경 (전체) | `changelog.md` | `updates.md` (선별본) |
| 앱 소개·가치 제안 | `overview.md` (개발자용), `store-description.md` (사용자용) | 서로 참조 없음 (관점이 다름) |
| 권한 설명 | `user-guide.md` | `faq.md` ("왜 이 권한이 필요해요?" 답변에서 링크) |
| 지원 플랫폼·최소 SDK | `changelog.md`의 "내부 구조" 섹션 | `store-description.md` (간단 명시) |

`changelog.md`가 "진실의 원천(source of truth)"입니다. 릴리스 시 `changelog.md`에서 공개할 항목을 골라 `updates.md`로 옮기고, `updates.md`의 최신 버전을 다국어로 가공해 `store-description.md`의 릴리스 노트 섹션을 갱신합니다.

## 5. 릴리스 유지보수 체크리스트

각 릴리스에서 아래 항목을 수행합니다.

- [ ] `changelog.md`에 버전 섹션 추가 (모든 변경 사항)
- [ ] 공개할 항목을 선별해 `updates.md`에 사용자 친화적 문구로 기재
- [ ] `store-description.md`의 "릴리스 노트" 섹션을 `updates.md` 최상단 버전 기반으로 갱신 (한/영)
- [ ] 기능이 추가·변경되었으면 `user-guide.md` 해당 섹션 수정 (스크린샷 포함)
- [ ] 사용자 문의에서 반복 질문이 생겼으면 `faq.md` 항목 추가
- [ ] 기술 스택이 바뀌었으면 `overview.md` 수정 (지원 플랫폼·최소 사양은 `changelog.md` 또는 `store-description.md`에 반영)
